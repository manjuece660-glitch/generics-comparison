Checked vs Unchecked Custom Exceptions — When to Use

Compact decision table and examples explaining when to design custom exceptions as checked (`Exception`) vs unchecked (`RuntimeException`).

Jump to table

Decision Table — When to use

Criterion Checked custom exception (extends Exception) Unchecked custom exception (extends RuntimeException)
Primary semantics Signals a recoverable / expected condition the caller should handle. Signals a programming error or an unexpected/unrecoverable condition.
Caller responsibility Compiler forces callers to handle (try/catch) or declare (throws) — explicit contract. No compile-time requirement; caller may choose to handle or ignore.
Typical use-case I/O errors, validation failures where caller can reasonably recover (e.g. FileFormatException, InsufficientFundsException). Illegal arguments, invariant violations, configuration faults, programming bugs (e.g. InvalidConfigurationException).
API boundary guidance Use when you want to force API users to think about handling the condition (library methods with recoverable failure). Use when exception indicates misuse or would add noise if forced to handle (frameworks often prefer unchecked).
Method signature void foo() throws MyCheckedException — part of the API contract. No throws required; document behavior instead.
Recoverability Usually recoverable; caller can retry, prompt user, choose alternative. Usually not recoverable at throw site — requires code fix or global handling.
Verbosity & ergonomics May cause many try/catch blocks — can harm API usability if overused. Cleaner call sites; less boilerplate.
Library trend Traditional Java libraries used checked exceptions (e.g., IOException). Modern frameworks prefer unchecked (Spring converts SQL exceptions to unchecked wrappers).
Example names FileFormatException, PaymentDeclinedException InvalidConfigurationException, MissingDependencyException
Quick decision: Checked → when caller must consciously handle the condition to continue safely. Unchecked → when it signals a programming / configuration error or would cause noisy client code.

Short examples

Checked custom exception 
// definition
public class FileFormatException extends Exception {
  public FileFormatException(String msg){ super(msg); }
  public FileFormatException(String msg, Throwable cause){ super(msg, cause); }
}

// usage (API)
public List parse(File f) throws FileFormatException {
  if (bad) throw new FileFormatException("malformed row 12");
  return result;
}

// caller
try {
  parser.parse(file);
} catch (FileFormatException e) {
  // recover: notify user, skip file, ask for fix
}
Unchecked custom exception 
public class InvalidConfigurationException extends RuntimeException {
  public InvalidConfigurationException(String msg){ super(msg); }
}

// usage
public void init(Config cfg) {
  if (cfg.get("host")==null) throw new InvalidConfigurationException("host is required");
}

Guidelines checklist

  • Can the caller reasonably recover? → prefer checked.
  • Is this a programmer/precondition error? → prefer unchecked.
  • Will forcing checked exceptions produce noisy client code with little benefit? → prefer unchecked or a single higher-level checked exception.
  • Document and use exception chaining (pass cause) when wrapping/transforming exceptions.
Best practice: choose a consistent policy for your library and document it. Use a small exception hierarchy (e.g. StorageException base) so callers can catch a logical group easily.
